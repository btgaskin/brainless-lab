A **task** chooses a world, one body per agent, rollout defaults, and optional scoring
metadata. A **world** owns external state and relations. A body owns sensing, encoding,
actuation, dynamics, and physiology. The reservoir is built from the body's resolved port
contract rather than from task-specific constants.

```text
TaskSpec ─setup→ TaskSetup(Environment, Bodies)
                         ├─ portspec(body) sizes each Reservoir
                         └─ Ensemble runs the synchronous lifecycle
```

`TaskSpec.n_receptors` and `n_effectors` are optional default metadata retained for
compatibility. Use `resolved_task_ports(task; task_options...)` when dimensions depend on a
composed setup. A task may set `score_key=nothing`; `:torus` is a first-class task without a
single scalar objective.

## Existing vector-valued task worlds

The established single-agent tasks already emit reservoir-shaped vectors. Their setup uses
a direct `Embodiment`: `DirectRelaySensor` + `IdentityEncoder` preserve the receptor vector,
and `DirectRelayActuator` returns the task-specific effector vector to the adapter.

| task | R | E | senses / acts with |
|---|---:|---:|---|
| `:wall` | 2 | 2 | two wall sensors; differential wheel vote |
| `:tracking` | 62 | 2 | two eyes × 31 bearing sensors; eye rotation |
| `:pong` / `:pong_hitrate` | 46 | 2 | ball-bearing bins; paddle vote |
| `:cartpole` (+ hard/swingup/long) | 8 | 2 | four state dimensions × two polarities; force vote |

Effector semantics are intentionally non-uniform: channel 1 in Pong is not channel 1 in
cartpole. `normalized_score` maps each task's raw metric between declared anchors, but it is
not a universal physical unit. See the [Task reference](/task-reference/) for operational
questions, controls, gates, and comparison limits; then read [Scoring](/scoring/) and
[Contracts](/contracts/).

## `ObjectWorld`: generic ecological composition

`ObjectWorld` is the generic physical environment for composed embodiments. It currently
supports:

- a `Torus` or `WalledArena`;
- fixed agent motion states and stable `EntityID` bindings;
- static circular object populations with stable `ObjectID`s;
- persistent, finite-capacity, and delayed-respawn resources;
- named analytic scalar fields;
- per-body `SpectralCamera` and `MountedFieldProbe` sampling;
- typed contact effects such as `Exposure(:energy, 0.2)`;
- one actuator/dynamics command pair per body.

```julia
include("examples/embodiments/object_world_task.jl")
sim = run_object_world_task(ticks=25, seed=11)

sim isa SimResult
getchannel(sim.recorder, :objects)
getchannel(sim.recorder, :poses)
```

`object_world_task.jl` is the tested, copy-ready `ObjectWorld` → `TaskSpec` → `SimResult`
path. Its setup callable accepts the standard `seed`, `rng`, `body`, and `n_nodes`
keywords. The smaller `object_world_quickstart.jl` demonstrates the lower-level path and
returns the live `Ensemble` and `Recorder` for direct inspection.

Object appearance is explicit. `AbstractObjectAppearance` is the extension boundary;
`NoAppearance()` is non-spectral, while `SpectralAppearance` couples a reflectance to camera
sampling. `rgb_appearance` is a convenient approximate RGB-to-spectrum constructor and
`spectral_reflectance` retrieves the optical surface. With `illuminant=nothing`, `ObjectWorld`
supplies a unit illuminant on each camera's own wavelength grid. Pass an explicit compatible
`SpectralIlluminant` when lighting itself is an experimental variable.

Object populations and analytic fields are independent. `ObjectType.bank` labels object and
render channels; it does not create a scalar field. A `MountedFieldProbe` reads the named field
explicitly supplied in `world.fields`; fields are not automatically centred on object
populations. Add or extend a field type when an experiment needs a source-centred gradient.

`sample_world_sensor!(sensor, world, motion_state)` is the extension seam for another
physical sensor kind. It returns that component's raw sample vector; the body's encoder
still owns the mapping to reservoir receptors. The `!` permits mutation of sensor or RNG
state; callers do not supply a destination buffer, and the contract does not promise zero
allocation.

Contacts produce `ObjectInteractionEvent`s and body effects. Query
`interaction_events(world)` and `object_snapshot(world)` without inspecting world storage.
`bind_entity_ids!` explicitly associates stable ensemble identities with world slots.
When several agents contact the same finite-capacity object in one tick, a seeded draw over
stable-ID-sorted contenders chooses the recipient. This is reproducible and independent of storage
grouping without permanently privileging the smallest identity.

Direct `ObjectWorld` composition exposes `Ensemble` + `Recorder`. Add a `TaskSpec` when the
experiment needs the standardized `SimResult`, rollout defaults, or scoring contract.
Entity-aligned channels such as `:poses`, `:receptors`, `:components`, `:needs`,
`:feedback`, `:body_alive`, and `:deaths` contain `EntityFrame`s with `.ids` beside their
values. `:interactions` and `:objects` contain world-global per-tick event vectors and object
snapshots.

### Current limits

`ObjectWorld` is deliberately small: objects are static circles, fields are analytic, and
populations are fixed. It does not yet provide moving non-agent entities, births,
replacement, lineage scheduling, arbitrary meshes, or a general event system. Physical
composition currently permits one actuator/dynamics command per body. Add ecological
complexity only where an experiment requires it.

`EmbodiedEnvironment` is the lower-level callback-driven physical runtime. Its sampler is a
callable `(body, motion_state, tick, arena) → raw samples`; the environment integrates the
body's command and projects motion into the arena. Use it for a focused custom sampler that
does not need `ObjectWorld`'s standard objects, fields, appearances, identities, or events.

## Multiple needs belong to physiology

An `Embodiment` can carry `RegulatedPhysiology` with any number of named
`RegulatedVariable`s. The world does not know a homeostatic equation: it emits typed
`Exposure`s when relations occur. The physiology applies matching changes, drift, bounds,
feedback, and failure.

This makes energy, hydration, temperature, security, or social contact composable without a
new world loop. A regulated variable defines its setpoint and deficit rule, response curve,
feedback mode, gain, stochastic emission probability, reservoir link probability, and
failure policy. Feedback remains off unless explicitly enabled.

All environmental effects for a tick are accumulated before physiology is updated. Failure
is then evaluated on the post-effect state, so a resource contact on a threshold tick can
rescue an agent. Once dead, an embodiment remains in its stable slot but no longer steps,
moves, senses, or interacts.

## The established situated adapter

`SituatedEnvironment` preserves the existing `:torus`, `:forage`, and signalling
behavior. It owns established bearing-bank assembly, collective
collision rules, signalling, histories, and their analysis channels. Bodies are still
ordinary `Embodiment`s, built with `SituatedSensorLayout`, `SituatedEncoder`,
`SituatedActuator`, and `KinematicMotor`.

This is an adapter, not the generic physical component runtime. Its historical default
collective bank uses 62 bearing rays with two reserved channels, yielding 64 receptors and
three kinematic effectors. `:forage` adds a source bank, yielding 128 receptors by default.
Colour-selective conspecific banks and acoustic signalling are adapter features.

```julia
simulate(:torus; node=:falandays_base, n_agents=12)

simulate(:forage;
    node=:falandays_base,
    n_agents=12,
    conspecific_vision=false,
)
```

The foraging comparison switches conspecific vision while preserving population size and
source sensing. Optional signalling adds a `:signal` effector and scalar `:acoustic`
receptor. Hearing is self-excluding, exponentially distance-attenuated, saturated, and
one-tick delayed by the synchronous sample/apply order.

`SituatedEnvironment` should be used when reproducing or extending those established tasks.
Use `ObjectWorld` when the experiment is about independently composed physical sensors,
actuators, dynamics, appearances, fields, and object effects.

## Writing a task

For a small vector-valued task, implement a `TaskWorld` (`sense`, `step!`, `reset!`,
`metrics`, dimensions, and defaults), wrap it in `TaskSpec`, and use a direct embodiment.
For an ecological task, construct bodies plus an `ObjectWorld` in the setup callable. In
both cases, keep the ownership test simple:

- world: external geometry, objects, fields, encounters, effects;
- body: physical components, receptor/effector semantics, physiology;
- reservoir: neural dynamics only;
- task: composition, defaults, and scoring metadata.

The copy-ready vector-task scaffold lives in `examples/templates/new_project/`; the three
embodiment presets live in `examples/embodiments/`.
