BrainlessLab is an extensible Julia lab for **"brainless" cognition**. It studies behaviour
that emerges from collectives of simple neuron-like nodes. These collectives have no trained
readout and no controller. The lab provides shared infrastructure for experiments around a
settled Falandays baseline. It is a Diverse Intelligences Summer Institute 2026 project.

For installation, an agent-assisted path, and the first runnable command, start with
[Getting started](/getting-started/).

## What the lab studies

BrainlessLab studies one class of entity from two angles. The [Concepts](/concepts/) page
describes this class. It is a self-organising collective, whether its units are nodes in a
reservoir or agents in a swarm.

The **cellular cognition** angle asks what this class can do. Each task operationalizes a
candidate capacity such as responsiveness or coordination. At the frontier, tasks can probe
adaptation and memory. The set can grow as tasks are added, but a task score is evidence of
task performance—not automatically evidence that the named capacity exists as a general
property. This approach is closer to construct validation than to a fixed benchmark.

A score is an operational indicator with declared anchors. The collective self-organises
toward its own viability, not toward our task. An honest result states what the score is
measured against, which controls were used, and what remains unvalidated. See
[Scoring](/scoring/) and the [Research workflow](/research-workflow/).

The **collective intelligence** angle asks what makes a capacity possible. It examines the
collective dynamics required for the capacity to appear, including criticality, timescale
separation, and coupling. The two angles meet because BrainlessLab measures the same dynamical
signatures at the node scale and the agent scale.

The body and world are experimental surfaces in their own right. A generic
[`Embodiment`](/receptors-effectors/) composes sensors, encoders, actuators, dynamics, and
physiology around any reservoir; [`ObjectWorld`](/environments-tasks/) composes the external
objects, fields, appearances, and effects that close the loop. The existing task suite uses
the same `AbstractBody` and `Ensemble` contracts through small adapters.

The [Platform limits](/platform-limits/) page states where these abstractions remain
simplified.

## One compute surface, optional views

`simulate` is the high-level compute surface and returns a `SimResult`. It runs headless;
Makie is an optional package extension. Load CairoMakie in a separate user or tool
environment for saved figures and animations, or GLMakie for the live `explore(...)`
window. The repository's root environment stays focused on the compute package.

The worked new-project template is also the checked-in visualization environment:

```bash
cd examples/templates/new_project
julia --project=. -e 'using Pkg; Pkg.develop(path="../../.."); Pkg.instantiate()'
julia --project=. run.jl --ticks 300 --seed 1
```

It saves `output/my_task_my_node_visualize.png`. The template project owns CairoMakie, so
these commands do not add plotting packages to BrainlessLab's root environment.

## The recording model

A run is only as observable as what it recorded. `simulate` returns a **`SimResult`**
(often bound to `sim`) — the recorded trace of a rollout, which the visualisation layer
reads *off* the hot path. What gets kept is gated by **record channels**: pass
`record=[…]` to select them, e.g. `record=[:effectors]` to keep the motor-output vector
each tick. The engine's only coupling to visualisation is a small channel-gated,
downsampling `Recorder`; `visualize`, `animate`, and `replay(sim)` all read the resulting
`SimResult`, and a saved run directory can be loaded back with `replay(rundir)`.

:::note[Not classical reservoir computing]
Recurrent spiking pools read out by simple effectors — but weights **adapt online**
(Falandays) or are **evolved** (compartmental), and there is **no trained readout**.
:::
