The **sensorimotor interface** — how many sensors an agent has, where they point, how
the world becomes receptor current, and how effector output becomes action — is a
first-class part of the experiment, not an implementation detail. This page is the
*mechanism and configuration* of that interface, plus where it is going. For the
per-task **R/E counts**, see [Environments & Tasks](/environments-tasks/); here we
cover how those counts are produced and which parts are now explicit specs.

## The interface

The reservoir talks to the world through two vectors: **receptors** $R$ (length
`n_receptors`, the sensory input) and **effectors** $E$ (length `n_effectors`, the
motor output). The **Body** owns the morphology that sits between them and the world:

```
percept ─receptors(body)→ R ─step!(reservoir,R)→ spikes ─effectors(reservoir)→ E ─decode_effectors(body)→ command ─actuate!(env)→ world
```

- **`PassthroughBody{M,Mt}`** carries a morphology and a motor. For single-agent tasks,
  the passthrough morphology relays the task env's $R$/$E$ unchanged — the `Env` owns
  `sense` / `step!` and task-specific effector decode.
- **The swarm body is `PassthroughBody(VENMorphology(...), KinematicMotor(...))`.**
  `VENMorphology` turns bearing vision into $R$ using a `SensorSpec`; `KinematicMotor`
  reads $E$ into movement dynamics. Coupling between agents is *vision*, so the sensor
  geometry literally *is* the interaction topology.

Effector semantics are **task-specific, not globally uniform** — channel 1 of Pong is
not the same physical quantity as channel 1 of cartpole. A reservoir is always built to
match the task/body it will drive, i.e. to the resulting `(n_receptors, n_effectors)`.
See the [node contract](/nodes/overview/) for what a reservoir must implement.

## Current state: task contracts plus explicit swarm specs

Single-agent task counts and placements still live in the task envs themselves. The
per-task numbers live in [Environments & Tasks](/environments-tasks/); the body just
passes those vectors through.

The swarm is now decomposed into three explicit pieces:

- **`VENMorphology`** owns the receptor/effector ports. The default torus layout is the
  historical conspecific bank: two reserved leads plus **62** bearing rays, so reservoirs
  are built `(n_receptors=64, n_effectors=3)`. `:forage` appends a second source bank using
  the same bearing geometry; with defaults that is `(128, 3)`. `signalling=true` adds a
  fourth `:signal` effector and reuses the first source reserved slot as `:acoustic`.
- **`BearingSensor <: SensorSpec`** owns the bearing geometry and encoding. `BearingSensor()`
  is the byte-identical historical two-eye 62-ray fan; `bearing_eyes(...)` and
  `BearingSensor(angles_deg=..., encoding=:graded)` let a run change the ray placement,
  active ray count, and binary/graded intersection encoding.
- **`KinematicMotor <: Motor`** owns the readout and kinematics. The default
  `readout=:spike_fraction`, `scheme=:ven_differential` path is the old VEN arithmetic;
  `readout=:graded_state` / `:graded_deviation` expose Falandays graded readouts, and
  `scheme=:ven_signed` can turn the thrust effector into signed forward/reverse drive.

Colour vision is a morphology option: `colour_sensing=true` lays out one conspecific
bearing bank per colour class (`conspecific_c0_bearing_...`, `conspecific_c1_bearing_...`).
The source bank stays uncoloured. With the defaults (`n_colours=1`, `colour_sensing=false`)
the receptor width and values are unchanged.

## Timing & temporal coding

Distinct from *what* the sensors are is *how the reservoir is clocked against the world*.

**Today, the Falandays path** runs exactly **one reservoir tick per env step**: graded
continuous input, effector read from that **single tick's** spike pattern. The "rate" is
the **spatial** proportion of the $N$ nodes spiking — a population code, not a temporal
accumulation. The reservoir is leaky, recurrent, and *not* reset between steps, so
temporal memory lives in its **state**; only the **readout** is instantaneous. This is
the stable clocking for `:falandays_base` (see [Falandays](/nodes/falandays/)).

**Where it bites:** fast tasks with a sensitive readout — **Pong** above all (fast ball,
jittery single-tick paddle command) — and small-$N$ runs.

**`substeps` (K) — now built for every node family.** The window is a first-class
trait ([`WindowTrait`](/nodes/overview/)): `IntrinsicWindow` nodes own their sub-stepping
inside `step!` (the compartmental CTRNNs — `CompartmentalReservoir(...; substeps=k)`,
**default 5**, $\text{dt}_\text{sub}=\text{dt}/5$, holding the afferent and reporting the
per-node spike *rate*); `SteppedWindow` nodes are single-tick maps that the framework loops
K times per env step, holding the afferent and mean-reducing the spikes. **Falandays is
now `SteppedWindow`** — `simulate(...; node=:falandays_base, node_kwargs=(substeps=K,))` (or
the `node.substeps` sweep axis). `substeps=1` (the default for every existing node/task) is
byte-identical to the previous single-tick readout.

Empirically this matters exactly where the single-tick readout is jittery: at N=200 the
Falandays baseline improves on **tracking** (≈0.28 → ≈0.80 from K=1 to K=5) and **pong**
(≈0.63 → ≈0.88), the tasks flagged below. It does **not** rescue cartpole — that task's
gap is sensorimotor (velocity-receptor scaling / bang-bang decode), not temporal averaging.

:::caution[Planned — the other readout-side timing knobs]
`substeps` above is built; the remaining levers still default to today's behaviour and
belong to the [remaining config/evolution layer](#remaining-roadmap-config-and-evolution) below:

| knob | default | meaning |
|---|---:|---|
| `input_encoding` | `:graded` | `:graded` (current value, held across the window) vs `:poisson` / `:regular` (spike train at rate ∝ value over the window). |
| `output_window` (W) | 1 | moving-average the effector over the last W env-steps — cheap, does not freeze the env, lags by ≈W. |

For Pong the env-preserving `output_window` (or a small `substeps`) beats a large `substeps`
that would freeze the ball. These belong to the same per-task spec as the sensor/effector
layout below.
:::

## Remaining roadmap: config and evolution

:::note[Roadmap, kept honest]
The swarm now has the core objects: `SensorSpec` / `BearingSensor` for geometry and
encoding, `Motor` / `KinematicMotor` for readout and kinematics, and `VENMorphology` for
ports. What is still a design target is the broader config surface and the morphology genome.
:::

Both roadmap angles start from the shipped split: **the morphology declares the ports, the
sensor declares the bearing geometry, and the motor declares the effector decode**.
`n_receptors` is the decoded input width after morphology padding, colour-bank expansion,
and source/signalling additions.

### Angle A — config-driven layout

Expose the shipped specs through TOML so an experimenter sets sensor **count, placement,
tuning, encoding**, colour vision, and motor readout/kinematics without touching code:

```toml
[swarm.sensor]
kind = "bearing"
encoding = "graded"
angles_deg = [-60, -40, -20, 0, 20, 40, 60]
tuning_deg = 0.0

[swarm.motor]
kind = "kinematic"
readout = "graded_state"
scheme = "ven_signed"
```

`resolve` would validate counts, angles, colour-bank widths, and motor compatibility, then
build the node to the resulting `(R, E)`. This is also where the timing knobs above would
live.

### Angle B — evolvable morphology (within set ranges)

Make part of the spec a **bounded morphology genome** that sep-CMA co-evolves *alongside*
the controller — "evolve where the sensors point, and how many," with researcher-set
ranges:

```toml
[task.sensors.evolve]
angle_range_deg = [-90, 90]     # each sensor angle bounded here
n_range         = [2, 32]       # variable length via fixed max + mask
tuning_range    = [4.0, 20.0]
```

Mechanics: a morphology parameter block is appended to the genome; each gene is mapped
through its bound, $g \mapsto \text{lo} + (\text{hi}-\text{lo})\,\sigma(g)$; the env
builds sensors from the decoded morphology each rollout. Variable sensor count is handled
by a fixed-max layout plus per-sensor enable gates. Controller and body then co-adapt —
the agent discovers its own sensorimotor arrangement. `BearingSensor` already has the
per-ray angle bounds, tuning bounds, and enabled mask that this layer would use; the missing
piece is connecting those fields to the optimizer genome.

## Why this matters for the collective

In the swarm the default coupling between agents is `BearingSensor` vision through
`VENMorphology` — so the sensor geometry (ray angles, field of view, range) **is** the
interaction topology. Making it tunable and evolvable means you can study how collective
behaviour (flocking, milling, foraging convergence) *depends on*, or *co-evolves with*, the
sensorimotor interface — not just the controller.
