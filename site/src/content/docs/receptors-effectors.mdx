An agent's embodiment is an experimental variable. Sensor placement, encoding, actuator
semantics, motion dynamics, and physiology all change the closed loop even when the
reservoir is identical. BrainlessLab therefore exposes one generic concrete body—
`Embodiment`—assembled from independently replaceable components.

## The public boundary

`AbstractBody` is the dispatch boundary. `Embodiment <: AbstractBody` is the standard
composition:

```julia
body = Embodiment(
    geometry=DiscGeometry(0.3),
    sensors=(left_probe, right_probe),
    encoders=(contrast_encoder,),
    actuators=(ForwardTurnActuator(0.8, 1.2),),
    dynamics=UnicycleDynamics(1.0, 0.1, 0.0),
    physiology=physiology,
    traits=(kind=:insect_like,),
    component_ids=(
        geometry=:body,
        sensors=(:antenna_left, :antenna_right),
        encoders=(:bilateral,),
        actuators=(:locomotion,),
        dynamics=:motion,
        physiology=:metabolism,
    ),
)
```

The same type represents biological and robotic organisms. The names in `traits` and
presets communicate an experimental interpretation; mechanics come only from the component
values.

## Stable components and ports

Every component has a stable `Symbol` ID. `component_slots(body)` returns identified
geometry, sensor, encoder, actuator, dynamics, and physiology slots. The body derives its
reservoir interface from the composed components:

```julia
spec = portspec(body)
n_receptors(spec)
n_effectors(spec)
ports(spec)
```

Port IDs are namespaced by the component that owns them, for example
`:bilateral__bilateral_1` or `:locomotion__turn`. This prevents collisions between repeated
component kinds and makes records and parameter overrides stable when tuple positions change.

Sensors declare a raw sample shape with `rawspec`; encoders turn one or more identified
sensor samples into receptor channels; actuators expose effector ports and decode reservoir
output into reusable typed command buffers. The body validates widths and references during
composition.

Configuration completes obvious identity mappings. If a TOML declares no encoder components,
materialization inserts one `:identity_encoder` over all sensors. If every declared encoder
names its sensor sources, each unclaimed sensor receives `<sensor>__identity_encoder`. Thus the
differential robot's camera needs no explicit encoder, while the UAV camera remains available
beside its explicit bilateral radio encoder. `component_slots(body)` shows the completed graph.

## The lifecycle

The complete path is:

```text
Environment.sample!
  → raw sensor samples
  → encode! for each encoder
  → physiology feedback receptors
  → Reservoir.step! and readout
  → decode! for each actuator
  → typed command buffer
  → integrate! through the dynamics component, or environment-specific action
  → environmental Exposure values
  → update! physiology
```

`sense!(body, percept)` returns the exact receptor vector delivered to the reservoir. For
`Embodiment`, this is a borrowed body-owned buffer that is overwritten by the next sensing
call; copy it if it must outlive the immediate reservoir step. The ensemble copies borrowed
vectors at its multi-agent gather boundary before another body is sensed. `decode!(body, values)`
mutates command buffers owned by the body's runtime state instead of
allocating a new command each tick. `component_state(body)` records state under stable
component IDs.

The built-in physical assembler currently accepts any number of sensors and compatible
encoders, but a physical body has one actuator/dynamics command pair. A body using
`NoDynamics` may instead be interpreted by an environment adapter.

## Sensors and encoders

The built-in component catalog includes:

- **`SpectralCamera`** — arbitrary named channels, ray geometry, mount, range, exposure,
  and channel sensitivity over a shared wavelength grid. Samples are flattened
  channel-major. Occlusion uses nearest circular ray hits.
- **`MountedFieldProbe`** — a scalar-field probe at a physical `Mount2D`, with first-order
  response lag plus separately seeded shared and independent Gaussian noise.
- **`BilateralFieldProbe`** — a direct Julia utility that samples one or more analytic
  fields at two explicit `Mount2D` positions. It returns channel-major left/right pairs.
  `SensorResponse` can then add response lag, shared noise, and independent noise. Reusable
  TOML bodies normally compose two `MountedFieldProbe` components instead.
- **`BilateralContrastEncoder`** — binds two stable sensor IDs and maps right-minus-left
  contrast into `[0, 1]`; equal inputs map to `0.5`.
- **`IdentityEncoder`** — a direct mapping for already encoded task vectors or sensors that
  should retain their raw channels.

Spectral colour is physical rather than an RGB tag lookup. A camera combines an illuminant,
target reflectance, and sensor sensitivity over a `SpectralGrid`; `rgb_appearance((r,g,b))`
is a convenient way to construct an approximate `SpectralAppearance`. `ObjectWorld`
samples `SpectralCamera` and `MountedFieldProbe` components per body. New sensor kinds extend
`sample_world_sensor!`.

The established `:torus` and `:forage` tasks use `SituatedSensorLayout` and
`SituatedEncoder` inside an ordinary `Embodiment`. That adapter preserves their bearing,
source, signalling, and colour-bank contracts while the generic physical component path
remains independent.

## Actuators and dynamics

Physical actuation is deliberately split in two:

| actuator | command | compatible dynamics |
|---|---|---|
| `ForwardTurnActuator` | target forward speed + turn rate | `UnicycleDynamics` |
| `DifferentialDriveActuator` | left + right wheel speed | `DifferentialDriveDynamics` |
| `PlanarForceYawActuator` | body-frame planar force + yaw torque | `PlanarRigidBodyDynamics` |
| `DirectRelayActuator` | bounded named vector | environment-specific or `NoDynamics` |

Effectors are normalized reservoir outputs; the actuator owns physical limits and reverse
policy, while the dynamics component owns integration, inertia, drag, and speed limits.
This keeps neural readout separate from vehicle mechanics.

## Physiology: configurable multiple needs

`RegulatedPhysiology` holds any number of `RegulatedVariable`s. Each variable owns its
bounds, initial state, setpoint/deficit rule, drift, response curve, feedback mode, gain,
emission probability, optional reservoir link probability, and failure policy.

Feedback is **off unless enabled**:

- `OffFeedback()` emits zero while physiology continues to change;
- `TonicFeedback()` emits the current scaled urgency;
- `BernoulliFeedback()` emits a stochastic pulse with deficit-scaled probability;
- `ReplayFeedback(values)` replays an explicit sequence for controls and interventions.

Built-in curves are constant, linear, power, logistic, and threshold. `Exposure(:energy,
0.2)` is an environmental effect: the world reports contact and the physiology applies the
matching delta. `RejectUnknownEffects()` is the safe default; use
`IgnoreUnknownEffects()` only when a mixed effect stream is intentional. Failure can be
disabled or triggered above/below a threshold; death is absorbing for the current body.

Physiology appends named feedback receptor ports after encoded sensory ports, so the final
reservoir dimensions still come entirely from `portspec(body)`.

## TOML composition

Embodiment files are strict, schema-versioned component graphs:

`traits` remain optional metadata for direct Julia composition. They are not currently
represented in embodiment TOML, and preset materialization or physical behavior does not
depend on them.

```julia
config = read_embodiment_config("examples/embodiments/bilateral_insect.toml")
blueprint = materialize_blueprint(config)   # typed components + stable IDs
body = materialize_embodiment(config)       # fresh runnable state
```

Each `[[components]]` entry declares `id`, `family`, `kind`, and a validated parameters
table. `extends` supports one base file; `[overrides]` targets stable component IDs and
parameter dot paths. Canonical serialization is available through
`canonical_embodiment_toml` and `write_embodiment_config`.

Three presets live under `examples/embodiments/`; see their
[README](https://github.com/btgaskin/brainless-lab/tree/main/examples/embodiments) for
materialization and runtime examples.

`examples/embodiments/object_world_quickstart.jl` is the executable end-to-end path. It
loads the differential robot preset, places an RGB beacon in an `ObjectWorld`, constructs a
Falandays reservoir from the resolved ports, binds a stable `EntityID`, records, steps, and
returns the ensemble plus object snapshot.

`examples/embodiments/object_world_task.jl` is the corresponding high-level path. It wraps
an `ObjectWorld` setup callable in a `TaskSpec` and returns a standardized `SimResult`.
Use the quickstart when direct access to the live `Ensemble` and `Recorder` is preferable.

## Component discovery and readiness

The component catalog is lightweight, typed discovery metadata:

```julia
components()                         # sorted descriptors
components(family=:sensor)
component_info(:sensor, :spectral_camera)
readiness()
readiness_markdown()
```

Descriptors publish required/optional parameter names, capabilities, documentation,
executable examples, and focused conformance evidence. Readiness has three ordered levels:

- `:available` — discoverable and materializable;
- `:integrated` — exercised through the standard runtime, exact serialization, docs, and an
  executable example;
- `:core` — a stable/default contract with named core-test coverage.

The built-in physical descriptors currently claim `:integrated` through the tested
`ObjectWorld` quickstart and exact configuration round trip. Status remains `:experimental`
at every readiness level. Registering an existing key
requires `replace=true`, so extensions cannot silently overwrite a component.

## Development is a bounded phenotype seam

`DevelopmentSpec` selects real scalar parameter paths on stable component IDs and gives
each a finite bound. A `DevelopmentGenotype` stores only those values. `develop` applies
them to a fresh component graph under a deterministic `DevelopmentContext`, and
`materialize_embodiment` produces a runnable phenotype.

Development does **not** mutate an existing body, encode transient state, add/remove
components, evolve topology, or schedule births. It is the bounded bridge from a fixed
configuration structure to fresh runnable embodiments. See [Evolution](/evolution/).
