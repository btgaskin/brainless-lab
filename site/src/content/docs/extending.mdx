title: Extending it
description: "Add a node, physical component, body, world sensor, task, metric, or analysis without forking the framework."

BrainlessLab's extension surfaces are Julia generics plus registries. Extend a contract,
register a discoverable constructor where appropriate, and compose it into an experiment.
Because you are adding methods to package functions, **import the names you extend**.

## Extension map

| You want to add | Start with | Copy-ready example | Minimum evidence |
| --- | --- | --- | --- |
| parameter variant of a node | config/preset or intervention | `configs/`, registered Falandays variants | resolved config plus regression |
| neural substrate | `Reservoir` generics | `examples/templates/new_project/my_node.jl` | widths, step, reset, replay, deterministic seed, task smoke run |
| drive or ablation | `Drive` or intervention registry | built-in drives/ablations | on/off mechanism test and recorded note |
| sensor/encoder | physical component contract | `examples/embodiments/*.toml` | geometry/orientation, exact port IDs and widths, world integration |
| actuator/dynamics | typed command pair | embodiment presets | limits, reverse/sign policy, integration, reset |
| physiology | `AbstractPhysiology` | regulated bilateral preset | drift/effect/feedback/failure timing and RNG ownership |
| body | composed `Embodiment` | three TOML presets | strict round trip and `Ensemble` smoke run |
| vector task | `TaskWorld` + `TaskSpec` | `examples/templates/new_project/my_task.jl` | static/random/oracle calibration, gates, reset, metrics |
| physical task | `ObjectWorld` + `TaskSpec` | `examples/embodiments/object_world_task.jl` | world randomization, effects, identities, task calibration |
| task metric | `metrics(environment, window)` | template task | raw definition, failure behavior, anchor provenance |
| reusable analysis | pure function + analysis registry | `skills/brainless-lab/references/designing-analyses.md` | channels, diagnostics, null, minimum data, finite-size check |
| optimizer/development seam | `AbstractEvolutionStrategy` or `DevelopmentSpec` | [Evolution](/evolution/) | pack/unpack, fresh phenotype, deterministic replay, search variance |
| visualization panel | view registry in the Makie extension | `ext/BrainlessLabMakieExt.jl` | headless core remains Makie-free; render test |

Prefer a parameter preset, composed value, or narrow method over a new type when the
behavioral contract has not changed. Register by name only when discovery or configuration
benefits from it.

## Add a node

First decide whether the change is truly a new substrate:

- a different constant bundle is a preset;
- a transient input is a drive;
- removing or clamping a mechanism is an intervention/ablation;
- different wiring behind the same update may be a constructor option;
- a different state/update/readout contract is a new `Reservoir`.

```julia
using BrainlessLab: Reservoir, register_node!, simulate
import BrainlessLab: step!, effectors, reset!, n_nodes, n_receptors, n_effectors

struct MyReservoir <: Reservoir
    # concrete fields
end

MyReservoir(n_nodes, n_receptors, n_effectors; seed=0, kwargs...) = # ...
step!(reservoir::MyReservoir, receptors) = # spike/rate vector
effectors(reservoir::MyReservoir, activity) = # effector vector
reset!(reservoir::MyReservoir) = # reset transient state
n_nodes(reservoir::MyReservoir) = # node population width

register_node!(:my_reservoir, MyReservoir; genome_type=MyParams)
simulate(:wall; node=:my_reservoir)
```

Use `pack_params` / `unpack_params` for evolvable parameters and
`snapshot_state` / `load_state!` for transient replay state. Do not mix them.

If the constructor accepts one input-connection probability per receptor, declare the
keyword as a registry capability rather than relying on its node name:

```julia
register_node!(
    :my_profiled_reservoir,
    MyReservoir;
    genome_type=MyParams,
    receptor_profile_keyword=:input_link_p,
)
```

Then an embodiment's `receptor_link_profile`—including each regulated variable's `link_p`—
is passed through that keyword. A node without the capability fails before rollout when a
body requests a non-uniform profile.

## Add a physical component

Choose the narrowest public family:

- sensor: subtype `AbstractSensor`, declare `rawspec`, and implement world sampling;
- encoder: subtype `AbstractEncoder`, declare `portspec`, implement `encode!`, and optionally
  declare `encoder_sources` by stable sensor ID;
- actuator: subtype `AbstractActuator`, declare ports and a reusable `command_buffer`, then
  implement mutating `decode!`;
- dynamics: subtype `AbstractDynamics` and implement `integrate!` for a compatible command;
- physiology: subtype `AbstractPhysiology` and implement the `physiology_*` contract;
- geometry: subtype `AbstractGeometry` and provide the physical queries your world uses.

A sensor that should work in `ObjectWorld` extends the sampling seam:

```julia
import BrainlessLab: rawspec, sample_world_sensor!

rawspec(sensor::MySensor) = (kind=:my_sensor, width=3)

function sample_world_sensor!(sensor::MySensor,
                              world::ObjectWorld,
                              state::MotionState2D)
    # Return a raw vector; `!` permits mutation of sensor or RNG state.
    return [0.0, 0.0, 0.0]
end
```

This contract does not take a caller-owned destination or promise zero allocation.

Register a strict config resolver with lightweight evidence metadata:

```julia
descriptor = ComponentDescriptor(
    :sensor,
    :my_sensor,
    resolve_my_sensor;
    capabilities=(:config_materialization, :my_capability),
    parameters=(required=(:channels,), optional=(:gain,)),
    conformance=:my_sensor_contract,
    conformance_path="test/test_my_sensor.jl",
    docs_path="docs/my_sensor.md",
    example_path="examples/my_sensor.toml",
    root=@__DIR__,
)

register_component!(descriptor)
```

Every evidence path must exist. Registration refuses duplicate keys unless
`replace=true` is explicit. Start at readiness `:available`; raise a claim only when the
corresponding integration/core evidence exists.

## Compose a body

Prefer constructing `Embodiment` to defining a new body subtype. Give every component a
stable, globally unique ID; let `portspec(body)` derive reservoir dimensions; and keep
runtime buffers inside component/body state rather than global dictionaries.

Use TOML when a composition should be reusable:

```julia
config = read_embodiment_config("my_embodiment.toml")
body = materialize_embodiment(config)
```

`examples/embodiments/` contains three complete presets. A custom `AbstractBody` is still
valid when the composition model is genuinely inappropriate; implement `sense!`, `decode!`,
widths, and optional lifecycle methods directly.

Materialization inserts one identity encoder when a TOML has no encoders. In a source-aware
composition, it also gives every unclaimed sensor a stable `<sensor>__identity_encoder`.
Inspect `component_slots(body)` before assuming that only explicitly listed encoders exist.

## Add an ecological task

For a physical task, construct bodies and an `ObjectWorld` in a `TaskSpec` setup. Use
`ObjectType`/`ObjectPopulation` for typed effects and appearance, named analytic fields for
continuous cues, and stable `EntityID`s for agents. Extend `sample_world_sensor!` rather
than teaching the world about an encoder.

Object banks do not generate fields: a mounted probe samples only the named analytic field
explicitly supplied in `world.fields`. The tested
`examples/embodiments/object_world_task.jl` shows a setup callable wrapped in `TaskSpec`,
with standardized `SimResult` recording. Use `object_world_quickstart.jl` when direct access
to the live `Ensemble` and `Recorder` is more useful.

For an already-vectorized single-agent task, implement the smaller `TaskWorld` contract:
`sense`, `step!`, `reset!`, `metrics`, receptor/effector counts, and defaults. Wrap it in a
`TaskSpec` and use a direct embodiment. The template under
`examples/templates/new_project/` is copy-ready.

Before selecting a node with the task, run static, random, blind/sham, and oracle/reference
policies; document warmup, randomized quantities, raw outcome, gates, degenerate policies,
and valid comparisons. See the [Task reference](/task-reference/).

## Add a metric or analysis

A **task metric** belongs to the environment and describes one rollout under that task.
Implement `metrics(environment, window)` and return a `NamedTuple`; the public fallback is
empty when an environment has no task-specific metrics.

A **reusable analysis** reads recorded channels from a `SimResult`. Before implementing it,
write down:

- required channels and their sampling cadence;
- node, agent, pooled, or world level;
- output and units;
- independent resampling unit;
- transient/window and missing/dead-run behavior;
- minimum data and estimator diagnostics;
- causal null or surrogate;
- threshold, bin, and finite-size sensitivity;
- known bias and unsupported interpretations.

Register it with an honest experimental label, add task relevance only where justified, and
test it on synthetic data with a known answer plus a failure case. Do not promote it because
it returns a plausible number. The [Analysis](/analysis/) page catalogs the built-ins; the
BrainlessLab skill's `designing-analyses.md` gives the full authoring contract.

## Add a drive, intervention, optimizer, or view

- A **drive** changes reservoir input/state through `apply_drive!`; seed and reset any
  stochastic stream and record the active drive.
- An **intervention** changes a declared mechanism and must be a logged no-op or explicit
  error when inapplicable. Pair it with a mechanism-specific control.
- An **optimizer** implements `ask`, `tell!`, and `result`; separate genome configuration
  from runtime state and measure search variance across independent optimization runs.
- A **view** belongs in the optional Makie extension, consumes recorded channels, and must
  not add plotting dependencies or computations to the simulation hot path.

## Other registries

| register | purpose |
|---|---|
| `register_task!` | a `TaskSpec` |
| `register_body!` | a reusable `AbstractBody` constructor |
| `register_drive!` | a reservoir drive |
| `register_metric!` | a rollout metric resolved by symbol |
| `register_analysis!` | a pure measure over recorded results |
| `register_ablation!` | a named intervention |
| `register_optimizer!` | an `AbstractEvolutionStrategy` |
| `register_view!` | an optional Makie panel |

Before adding a registry, check whether ordinary multiple dispatch and a composed value are
enough. Registries are for discovery and configuration, not a substitute for types.

## Verification checklist

- test the narrow component/body/world contract first;
- materialize from strict TOML, not only direct constructors;
- verify namespaced port IDs and exact receptor/effector widths;
- run through an `Ensemble`, including reset and recording;
- add a real example and point descriptor metadata to it;
- compare deterministic seeds and fresh-state behavior;
- warm hot calls before inference/allocation checks;
- update the canonical site page, README links, and relevant agent-skill reference;
- keep scientific claims separate from interface conformance.
