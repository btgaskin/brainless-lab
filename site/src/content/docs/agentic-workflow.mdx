title: Agent-assisted workflow
description: Use a coding agent to operate and extend BrainlessLab while preserving reproducibility and scientific control.
---

A coding agent can translate a research request into BrainlessLab types, plans, records, and
tests. The researcher still owns the question, scope, interpretation, and decision to
publish or promote evidence.

## Give the agent the repository context

The root `AGENTS.md` directs an agent to read:

- `skills/brainless-lab/SKILL.md` for platform concepts and scientific boundaries;
- `skills/julia/SKILL.md` for Julia design, testing, and package practice.

These files travel with the repository. The checked-in code, tests, plans, and records remain
the source of truth.

## State the request

A useful request contains:

```text
Outcome:
Scientific question:
What may change:
What must remain fixed:
Evidence state and seed stage:
Required checks and outputs:
```

You do not need to name Julia types. The agent should inspect the repository and select the
narrowest public interface.

Example:

```text
Outcome: add a three-channel directional sensor to a reusable body preset.
Scientific question: can a reservoir distinguish two source types from local observations?
What may change: one sensor or encoder, one preset, and one example.
What must remain fixed: Falandays fixtures and existing tasks.
Evidence state: mechanism conformance only; do not tune behaviour.
Required checks: port widths, orientation, reset, replay, a short ObjectWorld run,
package tests, and the site build.
```

## Use the safe change loop

1. **Inspect.** Read repository instructions and verify the live branch, worktree, code, and
   plans.
2. **Restate.** Name the outcome, scientific boundary, compute limit, and checks.
3. **Isolate.** Preserve unrelated user work. Use a branch or worktree for broad changes.
4. **Test the mechanism.** Verify ports, reset, replay, and local behaviour before a search.
5. **Implement the narrowest change.** Prefer composition and dispatch to node- or task-name
   branches.
6. **Check before running.** Validate typed plans without simulation.
7. **Run within the declared budget.** Keep development and held-out seeds separate.
8. **Verify.** Run focused tests, then package and site gates when the public interface
   changes.
9. **Report.** Separate software readiness, observed results, evidence state, and remaining
   limits.

For a broad change, an independent review can inspect architecture, use, and claim scope
without editing the implementation.

## Operate one run

Use `simulate` for a short diagnostic:

```julia
sim = simulate(:tracking; node=:falandays, ticks=1000, seed=11)
task_outcome(sim)
```

The agent should report the task, node, body, horizon, outcome key, raw value, normalised
value, and relevant warnings. One run is not an uncertainty estimate.

## Operate repeated research

Use a typed plan for a profile, sweep, ablation, evolution run, or benchmark:

```bash
julia --project=. bin/brainlesslab.jl check plans/examples/profile_tracking.toml
julia -t auto --project=. bin/brainlesslab.jl run \
  plans/examples/profile_tracking.toml --root records
```

The agent should show the resolved targets and compute budget before a large run. It should
not replace the plan with operation-specific command-line options.

Use `ExperimentSpec` when several named conditions and operations answer one scientific
question. `check-experiment` validates the complete protocol. `run-experiment` executes each
ordinary operation plan and records the protocol copy.

## Safeguards

The current platform provides:

- strict typed registries and duplicate-key rejection;
- pre-run composition, port, capability, and plan validation;
- fixture tests for the declared Falandays reference trajectories;
- named evaluation streams and a trial seed ledger;
- bounded sweep and evolution parameters declared by `NodeSpec`;
- registered ablations that fail when required capabilities are absent;
- request and resolved plan copies in every standard record;
- authoritative CSV tables, provenance, checksums, and completion markers.

These safeguards reduce implementation and bookkeeping errors. They do not validate a
research design automatically.

## Prompt patterns

### Design a task

```text
Turn this question into a TaskSpec proposal. Define observations, actions, interaction
timing, reset, horizon, randomised quantities, task outcome, descriptors, gates,
degenerate policies, nulls, and invalid comparisons before writing code.
```

### Add a node

```text
Decide whether this change is a parameter preset, ablation, drive, or new Reservoir.
Use NodeSpec and ParameterSpec when registration helps discovery. Keep runtime state out
of the genome. Add width, reset, replay, and short task tests.
```

### Run a sweep

```text
Resolve and show every cell before running. Use a SweepPlan, development seeds, paired
blocks, and a fixed rollout cap. Retain failed cells. Do not call the best cell an optimum
or inspect confirmation outcomes.
```

### Run evolution

```text
Resolve the node's named evolution parameter set. Use an EvolutionPlan with separate
training and held-out targets. Report candidate, search, and held-out records separately.
Do not infer transfer from the smoke-scale example.
```

### Prepare confirmation

```text
Audit the frozen ExperimentSpec without opening sealed outcomes. Check the primary endpoint,
paired randomisation, seed separation, sample-size basis, exclusions, nulls, limitations,
and required record artifacts.
```

## Human decisions

An agent should pause before it:

- changes the research question or primary endpoint;
- opens sealed evaluation outcomes;
- raises a search budget beyond the agreed limit;
- changes fixture expectations or reference semantics;
- deletes data or historical evidence;
- publishes, pushes, deploys, or changes an external system without authority.

BrainlessLab does not provide a permissions sandbox, automatic preregistration service,
sealed-data service, or automatic evidence-promotion command. Agent actions inherit the
permissions of their execution environment.

Use [Design a study](/core/design-study/) for evidence design and
[Platform limits](/platform-limits/) for the current automation boundary.
