A single-agent task and a twelve-agent swarm are the **same runtime at different `N`**.
Both are an `Ensemble`: agents in an environment, advanced through the same synchronous
`step!` lifecycle.

```julia
simulate(:torus;  node=:falandays_base, n_agents=2)    # dyad
simulate(:torus;  node=:falandays_base, n_agents=12)   # swarm
simulate(:forage; node=:falandays_base, n_agents=12)   # source foraging
```

## Synchronous, not sequential

For one tick, the runtime:

1. prepares the environment;
2. samples every body's percept from the same pre-action world;
3. senses and encodes receptors;
4. advances each reservoir and decodes commands;
5. applies the whole command frame;
6. updates physiology from the returned effects;
7. records the resulting entity/component state.

No agent observes another agent's already-committed motion from the same tick. This is the
meaningful invariant behind “one agent to swarm,” not simply reuse of a container type.

## Homogeneous and heterogeneous populations

If every agent has the same concrete type and port signature, `Ensemble` uses a homogeneous
concrete fast path. If agents differ in reservoir type, body type, or receptor/effector
width, it creates a grouped store. Each group remains concretely typed behind a function
barrier; only percept, command, and effect frames meet at the world-slot boundary.

The public accessors avoid exposing store representation:

```julia
nagents(ensemble)
entity_ids(ensemble)
agent_at_slot(ensemble, slot)
body_at_slot(ensemble, slot)

foreach_group(ensemble) do group
    group_agents(group)
    group_slots(group)
    group_ids(group)
end
```

`EntityID` is stable independently of slot and group. Recorded `EntityFrame`s carry IDs
beside their values so analysis does not silently infer identity from a vector index.

The population is currently fixed. Death keeps the identity and slot but makes the body
inactive; births, replacement, migration between runtime groups, and lineage events are not
implemented.

## Coupling lives in the world and sensors

The core ensemble adds no alignment, attraction, or social force. Coupling arises from
world relations:

- a generic `ObjectWorld` spectral camera may see circular resource objects;
- a field probe may sample a field changed by external structure;
- a situated bearing bank may see conspecifics;
- a collision or proximity relation may emit a body effect;
- an actuator changes world state that other agents sample on the next tick.

Sensor geometry therefore often **is the interaction topology**. Range, placement,
occlusion, response lag, noise, and encoding all change who can influence whom without a
separate graph in the controller.

`ObjectWorld` is the generic fixed-population physical runtime for independently composed
sensor/encoder/actuator/dynamics components. `SituatedEnvironment` is the established
adapter for `:torus`, `:forage`, signalling, and their historical analysis channels.
Current `ObjectWorld` spectral targets include object populations, not other agents; use the
situated adapter for conspecific visibility until entity targets are added to the generic world.

## The established torus and forage tasks

`TorusEnvironment` is periodic: distance, bearing, neighbour tests, and collective
centroids are wrap-aware. In the default task, agents influence one another through bearing
vision. `physical_coupling` optionally adds collision resolution.

`:forage` adds a stationary source and a separate source-sensing bank. The main comparison
switches only conspecific vision:

```julia
simulate(:forage; node=:falandays_base, n_agents=12, conspecific_vision=true)
simulate(:forage; node=:falandays_base, n_agents=12, conspecific_vision=false)
```

The blind condition preserves the number of agents and source sensing while zeroing the
conspecific bank. `physical_coupling` is an independent switch: collision resolution stays
on or off according to that setting in both visual conditions. The comparison asks whether
conspecific sensory information helps source approach; it does not create a separate
non-ensemble code path.

## Collective observables

The torus task has no scalar objective. Read it through order parameters and their nulls.

**Polarization** $P$ measures heading alignment:

$$
P = \left\lVert \frac{1}{N}\sum_i e^{i\theta_i} \right\rVert .
$$

**Milling** $M$ measures rotational order about the torus-aware centroid:

$$
M = \left| \frac{1}{N}\sum_i \left(\hat r_i \times \hat u_i\right) \right| .
$$

High $P$, low $M$ is a directed group; low $P$, high $M$ is a rotating mill. Other reads
include nearest-neighbour and pairwise distance, input stability, liveness, susceptibility,
correlation length, and contact-graph clusters. Forage adds distance to source, capture
fraction, first-arrival time, and its bounded task-specific score.

Cross-agent measures are vulnerable to shared input. Use the per-agent circular-shift null
(`crossshift_null`) before interpreting an aligned collective signature as interaction or
criticality. See [Analysis](/analysis/).

## Validation boundary

The authors-faithful Falandays node fixtures remain the scientific anchor for the baseline
reservoir. Established torus/dyad behavior also has implementation-oracle fixtures, but the
collective, physical embodiment, and ecological layers are experimental platform code—not
part of the original Falandays model.
