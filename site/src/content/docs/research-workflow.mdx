title: Research workflow
description: Move from a behavioral question to calibrated tasks, controlled comparisons, sealed confirmation, and traceable evidence.
---

BrainlessLab separates building a mechanism from establishing a scientific result. A
simulation that runs is an implementation result. A pattern that appears is an observation.
A claim needs a protocol whose controls, independent units, endpoints, and evidence status
match that claim.

## The evidence ladder

| State | Question | May influence later choices? | Suitable for a scientific claim? |
| --- | --- | --- | --- |
| **Conformance** | Does each component obey its stated mechanism? | yes | no |
| **Task calibration** | Can an oracle succeed, does a floor fail, and are timescales viable? | yes | no |
| **Exploration** | What behavior and failure modes appear? | yes | hypothesis generation only |
| **Tuning / training** | Which candidate parameters or genomes work on development worlds? | yes | no |
| **Variance pilot** | How variable is a frozen endpoint on fresh pilot blocks? | sample size only | no |
| **Frozen protocol** | Are conditions, endpoints, exclusions, and analysis fixed? | no | not yet |
| **Sealed confirmation** | Does the frozen contrast pass on untouched blocks? | no | yes, within scope |
| **Robustness** | Does the confirmed effect survive declared perturbations? | protocol-specific | supporting |
| **Promoted evidence** | Are protocol, code, seeds, data, and figures traceable? | immutable version | yes |

Looking at sealed outcomes and then changing a parameter, endpoint, exclusion, or analysis
returns the study to development. A new confirmation must use untouched blocks.

## Name the units

- **tick:** one synchronous world update;
- **agent:** one embodied reservoir inside a world;
- **trial/run:** one rollout from one complete randomized setup;
- **condition:** the intervention or model being compared;
- **cell:** one point in a sweep or factorial design;
- **block:** the shared randomized world within which conditions are paired;
- **seed:** an identifier used to create a random stream, not automatically an independent observation;
- **independent unit:** the smallest unit randomized independently for the inferential contrast.

Agents and ticks inside the same world are usually repeated observations, not additional
independent samples. A multi-agent trial with 20 agents is not automatically `n = 20`.

## 1. Turn the question into an operational task

Write down:

- the proposed capacity or mechanism;
- what the agent can observe;
- what each effector means;
- the primary raw outcome;
- task gates such as liveness or contact;
- known degenerate policies;
- randomized world quantities;
- reset, warmup, and episode semantics;
- the comparisons that would and would not be valid.

A task score operationalizes performance. It does not establish that the named capacity is
real or general. That interpretation comes from construct validity, controls, and
generalization.

## 2. Calibrate before tuning a reservoir

Use component tests and simple policies:

- a static or no-action policy;
- random action;
- a hand-written oracle or reference policy;
- deliberately blind or disconnected sensing;
- extreme starting states and boundary cases.

The oracle should repeatedly demonstrate the intended opportunity. Random action should not
trivially saturate the task. Travel, drift, restoration, and episode lengths should permit
several informative cycles. If failure or death is part of the task, a development regime
with almost 0% or almost 100% mortality is usually uninformative.

`calibration/` provides score anchors for existing tasks. Calibration establishes task
meaning; it is not reservoir selection.

## 3. Choose a control that matches the claim

| Control | What it asks |
| --- | --- |
| static/no-action | does doing anything help? |
| random action | is performance above this task/body/action floor? |
| blind or off | is the channel necessary relative to zero information? |
| sham, shifted, or yoked input | does informative timing or direction matter at matched dose? |
| mechanism ablation | is this implemented mechanism necessary? |
| model baseline | does a new substrate differ from the declared reference model? |
| oracle/reference | is the opportunity solvable and how much headroom remains? |

There is no universal null. `:null_random` is a task-, body-, and actuator-specific action
floor. It does not answer every causal question.

For paired comparisons, conditions in a block should share the declared world layout,
initial pose, and other nuisance randomization. Give distinct mechanisms their explicit
streams, and record the mapping. An exact replay is useful for regression; because it
reproduces the same input, it is not a causal null.

## 4. Explore and tune without contaminating evaluation

Use:

- `profile` to understand one node family;
- `sweep` to map a bounded parameter region;
- `ablate` to probe registered mechanisms;
- `evolve` or training only on declared development tasks and seeds;
- `benchmark` for a conditional roster-by-task comparison.

Cap factorial searches before running them. Inspect the resolved cells. Track every degree
of freedom that influenced selection: reservoir size, morphology, source layout, vision
range, gain, noise, drift, restoration, distance, duration, and failure policy all count.

Keep disjoint seed ledgers for:

- task and mechanism conformance;
- exploratory tuning or training;
- variance pilot;
- sealed confirmation;
- robustness.

## 5. Freeze endpoints and plan independent replication

Declare a smallest meaningful effect, not merely a desired p-value. Estimate paired
variability on fresh pilot blocks, then calculate the required number of independent blocks
prospectively. If the required sample exceeds the resource cap, simplify the question or
report that it is underpowered; do not label a capped run confirmatory.

Bootstrap or permutation procedures must resample the independent unit. Resampling agents or
ticks as though they were independent creates false precision. Retrospective “achieved
power” is not a substitute for prospective planning.

For a claim of no meaningful difference, define an equivalence margin and use an equivalence
procedure. A confidence interval that crosses zero, or a nonsignificant test, does not
establish equivalence.

## 6. Analyze with declared discipline

Before seeing confirmation outcomes, fix:

- primary and secondary endpoints;
- transient removal and analysis window;
- missing, failed, or dead-run handling;
- estimator diagnostics and minimum-data rules;
- threshold and binning sensitivity where relevant;
- multiplicity handling;
- uncertainty interval and resampling unit;
- null or surrogate for each causal or dynamical claim.

Criticality, information-flow, and collective-order statistics are descriptors. They should
be read alongside task performance and their relevant nulls, not promoted to competence
scores.

## 7. Promote evidence deliberately

A promoted study bundle should contain:

- a frozen protocol and analysis plan;
- resolved configuration and selected parameters;
- full git SHA and dirty-worktree status;
- Julia version plus Project and Manifest hashes;
- a named seed ledger with cross-stage overlap checks;
- per-block data and declared paired contrasts;
- inferential unit, exclusions, and failure handling;
- analysis-code version;
- schema-versioned summary data;
- figure inputs and representative-selection rule;
- checksums;
- an external archive URI and hash when raw data is too large for Git.

Figures and prose should read values from the promoted summary rather than copy numerical
claims by hand. Exploratory runs may be useful and beautiful without being promoted.

## Agent-assisted research checklist

Before asking an agent to summarize or extend a result, require it to:

1. identify the evidence status;
2. verify the artifact path, git SHA, and dirty flag;
3. name the independent unit and number of independent blocks;
4. check seed-stage overlap;
5. distinguish tuned selection from sealed evaluation;
6. match each claim to its control and null;
7. leave sealed data unopened unless executing the frozen protocol;
8. report limitations and failed gates with the result.
