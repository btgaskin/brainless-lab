Evolution is experimental platform infrastructure. Some reservoirs adapt online and can be
run directly; fixed-weight reservoirs are not meaningful comparisons until evolved.
Embodiment parameters can now join a search through a bounded development plan, without
turning runtime state into a genome.

## Which reservoirs need evolution?

| family | online plasticity | evolution |
|---|---|---|
| `:falandays` (`:falandays_base` compatibility alias) | yes | optional; defaults are the settled authors-faithful baseline |
| experimental Falandays variants | generally yes | optional experimental perturbation |
| `:compartmental_dense`, `:compartmental_structured` | no | required; untrained weights are random |

The baseline's authors-faithful fixtures remain fixed. Evolving its control parameters is a
new experiment, not a replacement baseline.

## Reservoir optimizer

The built-in optimizer is separable CMA-ES (diagonal covariance), minimizing
`loss = -fitness`. Independent rollouts are thread-parallel and returned in deterministic
seed order. Within one generation, candidates share common-random-number trial seeds so
candidate differences are not confounded by different task draws.

For each task, fitness is the mean normalized score over `k_trials`. A multi-task run then
aggregates across tasks with `:min` (default worst-task fitness) or `:mean`. Normalized
scores use each task's own anchors and remain task-specific indicators; see
[Contracts](/contracts/).

```julia
result = evolve(
    model_sym=:compartmental_structured,
    train_tasks=(:wall,),
    generations=30,
    popsize=32,
    k_trials=8,
    N=200,
)
```

`find_alive_centroid` can initialize searches away from a dead region of fixed-weight
genome space. The benchmark genotype store records the genome alongside its training
manifest and content tag so evaluation artifacts identify the exact trained parameters.

## Bounded embodiment development

`DevelopmentSpec` connects a fixed embodiment configuration to an evolvable vector. Each
`DevelopmentBlock` targets real scalar parameter paths on one stable component ID and gives
each path finite bounds:

```julia
config = read_embodiment_config("examples/embodiments/differential_robot.toml")

spec = DevelopmentSpec(
    config,
    DevelopmentBlock(
        :camera_genes,
        :camera,
        "range" => (2.0, 15.0),
    ),
    DevelopmentBlock(
        :wheel_genes,
        :wheels,
        "max_speed" => (0.2, 2.0),
    ),
)

genotype = DevelopmentGenotype(spec)
context = DevelopmentContext(seed=7, entity_id=EntityID(41), generation=3)
blueprint = develop(genotype, context)
body = materialize_embodiment(blueprint)
```

Paths may cross fixed tuples by one-based index (`"mount.2"`) and collections whose entries
carry stable names (`"variables.energy.gain"`). Named need paths remain stable if the tuple is
reordered; numeric paths such as `"variables.1.gain"` are available when position is the
intended genotype contract.

`paramspace(spec)` exposes labeled bounds and `pack_params(spec)` returns initial values.
`unpack_params`, `mutate`, and `recombine` preserve bounds and require the same structural
signature. `composite_genome(spec)` adapts development blocks to the existing composite
genome machinery.

The context derives stable per-component/per-stream seeds from the experiment seed,
`EntityID`, generation, component ID, and stream name without Julia's randomized `hash`.
This includes field-probe shared/independent noise and regulated-physiology feedback noise.
Every development event materializes fresh component state; it never carries a previous
body's lag buffers, physiology, commands, or noise position into a new phenotype.

## What development does not encode

The current seam is intentionally bounded:

- component families, IDs, ordering, and references stay fixed;
- genes target existing real scalar TOML parameters only;
- actuator/dynamics compatibility is validated before search;
- bilateral encoders must reference two existing distinct sensor IDs;
- every runnable standard embodiment requires at least one sensor and actuator, with
  geometry, dynamics, and physiology each singular;
- physical development additionally requires exactly one geometry, actuator, and dynamics
  component; multiple actuators are only valid for a non-physical composition with no
  dynamics component;
- component addition/removal, variable sensor count, graph topology, runtime births,
  replacement, and lineage scheduling are outside the contract.

That is enough to evolve sensor range/exposure, physical placement coordinates, response
times, actuator limits, dynamics constants, or physiological gains while preserving a
clean, reproducible phenotype structure. Structural evolution can be added later as a
separate operation rather than smuggled into scalar parameters.

## Genotype is not state

Keep three representations distinct:

1. **genome** — parameters an optimizer stores and mutates;
2. **developed blueprint** — resolved component structure plus deterministic development
   context;
3. **runtime state** — membrane values, learned weights, sensor lag/noise state, command
   buffers, physiology, and motion state.

For reservoirs, `pack_params`/`unpack_params` describe the genome and
`snapshot_state`/`load_state!` describe transient replay state. For embodiments,
`DevelopmentSpec`/`DevelopmentGenotype` describe the genome and
`materialize_embodiment(develop(...))` constructs a fresh phenotype.

## Statistical honesty

A trained cell evaluated over many seeds measures evaluation variance for one selected
genome. It does not measure search variance. Strong comparisons should run multiple
independent evolutionary searches and report the distribution across selected genomes,
then evaluate each on untouched seeds.
