`EvolutionPlan` is the supported operation for parameter evolution. It selects one
registered node parameter set on a training target. It can then evaluate the selected
champion on fresh seeds or other tasks.

Evolution changes a declared parameter vector. It does not change the meaning of the
reference preset, and it does not establish generality from one search.

## Declare what may evolve

Each node uses `ParameterSpec` to define its configurable values. A parameter can include
bounded or categorical evolution metadata. `NodeSpec.parameter_sets` gives a stable name to
the group that an operation may search.

For example, the Falandays `:evolve` set currently includes its registered local-rule and
initialisation parameters. `EvolutionPlan` resolves this set from the registry. It does not
infer evolvable values from reservoir fields.

Node count belongs to `CompositionSpec`, so it is fixed by this operation. Structural
evolution also remains outside `EvolutionPlan`.

## Write an evolution plan

An evolution plan has one training target and zero or more held-out targets:

```toml
format = "brainlesslab-plan"
format_version = 1
operation = "evolve"
id = "evolve_tracking_test_pong"

[[targets]]
id = "tracking_development"

[targets.composition]
id = "falandays_tracking_evolution"
preset = "falandays_tracking"

[targets.evaluation]
blocks = 4
trials_per_block = 2
horizon = 7200
warmup = 100
construction_scope = "trial"
reset = "full"
root_seed = 1401
aggregate = "mean"

[[targets]]
id = "pong_heldout"

[targets.composition]
id = "falandays_pong_heldout"
preset = "falandays_pong"

[targets.evaluation]
blocks = 4
trials_per_block = 2
horizon = 100000
warmup = 100
construction_scope = "trial"
reset = "full"
root_seed = 2401
aggregate = "mean"

[evolve]
training = "tracking_development"
heldout = ["pong_heldout"]
optimizer = "sepcma"
parameter_set = "evolve"
objective = "normalized_score"
generations = 50
popsize = 64
sigma0 = 0.5
```

Use `check` before the run:

```bash
julia --project=. bin/brainlesslab.jl check path/to/evolution.toml
julia -t auto --project=. bin/brainlesslab.jl run \
  path/to/evolution.toml --root records
```

Validation requires every target to use the same registered node. Each target must use
`reset = "full"` and a scalar aggregation policy. Held-out tasks must accept every evolved
parameter.

## What the operation does

The registered `:sepcma` optimiser proposes a population in each generation. Candidate
evaluations use the training target only. Within one generation, candidates share the same
trial protocol and seed derivation.

The optimiser random stream is derived separately from evaluation streams. Candidate
evaluations run in parallel when Julia has more than one thread, but output order and seed
records remain deterministic.

After the final generation, the operation selects one champion. It evaluates that champion
on the training target and then on each held-out target. Held-out outcomes do not influence
selection.

The result contains:

- every candidate's coordinates, resolved parameters, trial values, and fitness;
- one convergence row per generation;
- the champion parameters and training fitness;
- fresh training and held-out evaluation batches;
- the optimiser seed and complete trial seed ledger.

The generated HTML report includes convergence and champion comparisons. CSV remains the
authoritative tabular output.

## Choose an objective carefully

`objective = "normalized_score"` uses the task's declared score anchors.
`objective = "raw_score"` uses its raw outcome. A plan can also name the task outcome key or
a numeric metric field.

The evaluation's aggregation policy combines trial values for one candidate. This policy
must match the scientific question. Normalised scores remain task-specific, even when they
share a numeric range.

## Separate selection from evidence

Training evaluations are development data. A held-out target in the same operation protects
that target from direct selection, but one evolutionary run still gives only one selected
genome.

A stronger study should:

1. run several independent searches;
2. preserve the distribution of selected genomes and search outcomes;
3. freeze the selection procedure before confirmation;
4. use untouched confirmation blocks;
5. report both search variance and evaluation variance.

The reciprocal Tracking and Pong example is a useful demonstration design: evolve on
Tracking and test on Pong, then reverse the direction. Small checked-in versions test the
operation path. They are not evidence about transfer.

## Bounded embodiment development

`DevelopmentSpec` is a lower-level way to expose bounded scalar parameters from a fixed
embodiment graph. Stable component IDs and parameter paths identify each value.

It keeps genome, developed blueprint, and runtime state separate. Development currently
does not add components, vary sensor count, or evolve graph structure. A future operation
can compose this facility with a study protocol without changing the node-parameter
meaning of `EvolutionPlan`.

## Legacy optimiser API

The package still exposes the lower-level `evolve(...)` driver and `DevelopmentSpec`
machinery for specialised Julia workflows. That API predates typed operation plans and has
different multi-task and genome options.

Use it when you need those low-level facilities. Do not treat its keyword interface or
output tuple as the record-producing experiment path. New reproducible studies should use
`EvolutionPlan`, `EvaluationSpec`, and the standard record writer.
