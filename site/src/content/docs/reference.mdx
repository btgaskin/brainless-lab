This page lists the stable entry points for current platform work. Query the registries
instead of copying a static list of built-in names.

## Discover registered items

`DEFAULT_REGISTRY` contains nodes, tasks, bodies, analyses, optimisers, ablations, and named
composition presets.

```julia
using BrainlessLab

nodes(DEFAULT_REGISTRY)
tasks(DEFAULT_REGISTRY)
tasks(DEFAULT_REGISTRY; tag=:benchmark)
analyses(DEFAULT_REGISTRY; task=:tracking)
ablations(DEFAULT_REGISTRY)
compositions(DEFAULT_REGISTRY)
```

Resolve one descriptor:

```julia
node = node_spec(DEFAULT_REGISTRY, :falandays)
task = task_spec(DEFAULT_REGISTRY, :tracking)
preset = composition_spec(DEFAULT_REGISTRY, :falandays_tracking)
```

Other typed registries are available as fields of `DEFAULT_REGISTRY`:

```julia
sort!(collect(keys(DEFAULT_REGISTRY.bodies)); by=string)
sort!(collect(keys(DEFAULT_REGISTRY.optimizers)); by=string)
```

The configured embodiment component catalogue is separate:

```julia
components()
components(family=:sensor)
component_info(:sensor, :spectral_camera)
readiness()
```

Component readiness describes software integration. It does not establish scientific
evidence.

## CompositionSpec

`CompositionSpec` is one serialisable node, task, and body composition:

```julia
composition = CompositionSpec(
    :tracking_example,
    :falandays,
    :tracking;
    n_nodes=200,
    parameters=Dict(:leak => 0.25),
)

resolved = resolve_composition(composition, DEFAULT_REGISTRY)
```

Fields:

| Field | Meaning |
| --- | --- |
| `id` | stable name for this composition |
| `node` | registered node key |
| `task` | registered task key |
| `body` | optional registered body key |
| `n_agents` | optional task population override |
| `n_nodes` | node count per reservoir |
| `parameters` | declared node parameter overrides |
| `task_options` | task setup options |
| `body_options` | registered body options |
| `interaction_cycle` | optional override for neural frames per world step |

`default_composition(DEFAULT_REGISTRY, node, task)` returns the registered default for a
node and task pair when one exists.

## EvaluationSpec

`EvaluationSpec` is the only outer repetition protocol:

```julia
evaluation = EvaluationSpec(
    blocks=4,
    trials_per_block=2,
    horizon=7200,
    warmup=100,
    construction_scope=:trial,
    reset=:full,
    root_seed=4101,
    aggregate=:mean,
)

target = EvaluationTarget(:tracking, composition, evaluation)
```

Allowed construction scopes are `:evaluation`, `:block`, and `:trial`. Reset policies are
`:full`, `:body_environment`, and `:none`. Aggregation policies are `:none`, `:mean`,
`:median`, `:sum`, `:minimum`, and `:maximum`.

Default named streams are `:topology`, `:node_state`, `:world`, `:body`, `:task`, and
`:mechanism`. Use `SeedStreamSpec` to declare another stream and `derive_seed` to derive a
stable seed.

## Operation plans

All plan files use:

```toml
format = "brainlesslab-plan"
format_version = 1
operation = "profile"
id = "my_plan"
```

They then contain one or more `[[targets]]` tables and one operation section.

| Operation | Type | Section |
| --- | --- | --- |
| profile | `ProfilePlan` | `[profile]` |
| sweep | `SweepPlan` | `[sweep]` |
| ablation | `AblationPlan` | `[ablate]` |
| evolution | `EvolutionPlan` | `[evolve]` |
| benchmark | `BenchmarkPlan` | `[benchmark]` |

Read, check, resolve, and write plans with:

```julia
plan = read_plan("path/to/plan.toml")
validate(plan, DEFAULT_REGISTRY)
resolved = resolve(plan, DEFAULT_REGISTRY)
write_plan("path/to/copy.toml", plan)
```

Run the corresponding CLI:

```bash
julia --project=. bin/brainlesslab.jl check path/to/plan.toml
julia -t auto --project=. bin/brainlesslab.jl run path/to/plan.toml --root records
```

See [Operations and records](/core/operations-records/) for the complete workflow.

## ExperimentSpec

`ExperimentSpec` groups named conditions and operation plans under one scientific question:

```julia
experiment = ExperimentSpec(
    :my_experiment,
    v"1.0.0";
    title="My experiment",
    question="How does the declared change affect tracking?",
    conditions=(target,),
    operations=(plan,),
    evidence_state=:planned,
    limitations=("Smoke-scale example only.",),
)
```

Allowed evidence states are `:planned`, `:exploratory`, `:tuned`, `:frozen`, `:confirmed`,
`:promoted`, and `:retired`.

```julia
write_experiment("experiments/my-experiment", experiment)
loaded = read_experiment("experiments/my-experiment")
register_experiment!(loaded)
experiments()
experiment_spec(:my_experiment, v"1.0.0")
```

Repeated condition names must have identical definitions in every operation. Declared
conditions must be used.

## Embodiment configuration

Embodiment TOML uses schema version 1:

```toml
schema_version = 1
name = "my_embodiment"

[[components]]
id = "camera"
family = "sensor"
kind = "spectral_camera"

[components.parameters]
range = 10.0
```

Use:

```julia
config = read_embodiment_config(path)
canonical_embodiment_toml(config)
write_embodiment_config(output_path, config)
body = materialize_embodiment(config)
```

Component IDs must be unique and cannot contain `.`. Unknown keys and parameters fail
validation.

## Results and outcomes

`simulate` returns `SimResult`. Use:

```julia
outcome = task_outcome(sim)
```

The result is `nothing` when the task declares no scalar outcome. Otherwise it contains the
task key, raw value, and normalised value. Inspect the resolved `TaskSpec` when you also need
the anchor values and provenance.

Typed operation results support:

```julia
tables(result)
summary(result)
```

The record writer stores authoritative CSV tables, a compact JSON summary, the submitted
and resolved plans, provenance, checksums, and a generated HTML report.

## Optional inspection hooks

`network_snapshot(reservoir)` returns read-only network metadata when a node supports it.
`component_state(body)` returns state keyed by stable component IDs. These hooks support
recording and visualisation; they are not parameter genomes or replay snapshots.
