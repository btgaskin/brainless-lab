## Embodiment configuration

```julia
config = read_embodiment_config(path)
embodiment_config_namedtuple(config)
canonical_embodiment_toml(config)
write_embodiment_config(output_path, config)

blueprint = materialize_blueprint(config)
body = materialize_embodiment(config)
```

Schema version: `EMBODIMENT_SCHEMA_VERSION == 1`.

Top level:

```toml
schema_version = 1
name = "my_embodiment"
# extends = "base.toml"      # optional one-level base

[[components]]
id = "stable_id"
family = "sensor"
kind = "field_probe"
[components.parameters]
# kind-specific validated parameters

[overrides]
# "stable_id.parameter.path" = value
```

Generic families are `geometry`, `sensor`, `encoder`, `actuator`, `dynamics`, and
`physiology`. IDs are globally unique and may not contain `.` because dot paths address
parameters. Unknown keys and parameters are errors.

Checked-in presets:

| file | composition |
|---|---|
| `examples/embodiments/bilateral_insect.toml` | two field probes → contrast encoder; forward/turn + unicycle; energy and temperature physiology |
| `examples/embodiments/differential_robot.toml` | three-channel spectral camera; differential actuator + dynamics |
| `examples/embodiments/planar_uav.toml` | spectral camera + paired radio probes; planar force/yaw + rigid-body dynamics |

Executable physical example:
`examples/embodiments/object_world_quickstart.jl` →
`run_object_world_quickstart(; ticks=25, seed=7)`. It returns the live `Ensemble`,
`Recorder`, and object snapshot.

Executable task example:
`examples/embodiments/object_world_task.jl` →
`run_object_world_task(; ticks=25, seed=11)`. It wraps a generic setup callable in
`TaskSpec` and returns a standardized `SimResult`.

## Built-in configured components

Query accepted parameter names rather than copying constructor fields:

```julia
component_info(:sensor, :spectral_camera).parameters
components(family=:actuator)
components(readiness=:integrated)
readiness_markdown()
```

The catalog reports required/optional names only; it is not a typed schema of defaults or
constraints.

The built-in configured kinds are:

- `:geometry/:disc`
- `:sensor/:spectral_camera`, `:sensor/:field_probe`
- `:encoder/:bilateral_contrast`
- `:actuator/:forward_turn`, `:actuator/:differential_drive`,
  `:actuator/:planar_force_yaw`
- `:dynamics/:unicycle`, `:dynamics/:differential_drive`,
  `:dynamics/:planar_rigid_body`
- `:physiology/:regulated`

Readiness levels are ordered `:available`, `:integrated`, `:core`; the filter is a minimum.
Available means discoverable/materializable; integrated adds standard runtime, exact
serialization, docs, and an executable example; core is a stable/default contract with named
core-test coverage. The built-in physical catalog currently claims `:integrated` through
the tested `ObjectWorld` path. Every descriptor still has `status=:experimental`.

## Generic world surface

`ObjectWorld` composes `Torus`/`WalledArena`, `MotionState2D`s, `ObjectPopulation`s, named
analytic fields, and an optional spectral illuminant. Its public inspection/identity surface
is:

```julia
bind_entity_ids!(world, ids)
interaction_events(world)
object_snapshot(world)
```

Objects use `ObjectType` with optional `appearance`, capacity, effects, and respawn policy.
`AbstractObjectAppearance` is the extension boundary. Built-ins are `NoAppearance` and
`SpectralAppearance`; `rgb_appearance` provides a convenient spectral approximation. The current world is fixed-population with static
circular objects and one physical actuator/dynamics command per body.

Object banks and analytic fields are independent: assigning an object `bank` does not emit a
field. `MountedFieldProbe`s sample only fields explicitly present in `world.fields`. With no
explicit illuminant, spectral cameras receive a unit illuminant on their own wavelength grid.

## Registered tasks

`:wall`, `:tracking`, `:pong`, `:pong_hitrate`, `:cartpole` (+ `_hard`, `_swingup`,
`_long`), `:torus`, and `:forage`.

The generic physical `ObjectWorld` is a composition surface, not an additional default task
symbol. Wrap a setup in a local or registered `TaskSpec` when standardized rollout and result
semantics are needed.

`RngNoise(seed)` passed through the high-level multi-agent builder is materialized as an
independently owned stream per agent with seeds `seed`, `seed + 1`, and so on. Use
`AgentNoiseFactory(index -> source)` when a custom noise source needs an explicit construction
rule; custom source types may instead extend `agent_noise_source`.

## Development

```julia
block = DevelopmentBlock(
    :sensor_genes,
    :camera,
    "range" => (2.0, 12.0),
)
spec = DevelopmentSpec(config, block)
genotype = DevelopmentGenotype(spec)
context = DevelopmentContext(seed=0, entity_id=EntityID(1), generation=0)
body = materialize_embodiment(develop(genotype, context))
```

Discovery/helpers: `paramdim`, `paramspace`, `pack_params`, `unpack_params`,
`development_seed`, `validate_development_structure`, and `composite_genome`.

## Sweep config schema

```toml
[sweep]     # id, mode = one_at_a_time | factorial, seeds, max_cells
[baseline]  # node, task, N, ticks and fixed dotted options
[axes]      # "node.x" / "env.x" / "drive.x" / "task.x" / "ablation" / "seed"
[analytics] # measures = [...]
```

Run `sweep --list-axes <node> <task>` for resolved axes. Common baseline keys are `node`,
`task`, `N`/`n_nodes`, `ticks`, `window`, `n_agents`, `seed_base`/`seed`, `body`, `drive`,
and `ablation`. Dotted environment and node options pass through to the run builder.

Common measures include `sigma_mr`, `sigma_mr_node`, `sigma_mr_agent`,
`spectral_radius`, `susceptibility_node`, `susceptibility_agent`, `correlation_length`,
`contact_clusters`, `dist_to_source`, `forage_score`, `liveness`, and `regime`. See
[Tooling](/tooling/) for the workflow and [Analysis](/analysis/) for interpretation.
