title: Scoring
description: What a score means for a self-organising system — calibrated anchors, honest nulls, and gates that keep an indicator from being gamed.
---

A score here is a **task-specific operational measure of performance**—never the capacity
itself and never sufficient evidence that a capacity is present. A self-organising
collective maintains its own *viability*; it has no intrinsic reason to play Pong. We
measure what it does under an explicit task contract, and a bare number means nothing until
we say what it is measured *against*. Each task's raw metric can be mapped onto $[0,1]$
against two **anchors**, a floor and a ceiling:

$$
\text{normalised} = \operatorname{clamp}\!\left(\frac{\text{raw} - \text{floor}}{\text{ceiling} - \text{floor}},\; 0,\; 1\right)
$$

Read it as *the fraction of this task's declared anchor gap that the run closed*. Zero is
not necessarily “did nothing”; it is performance at or below this floor.

## Anchors carry their provenance

`ScoreAnchor` replaces the hand-typed magic numbers. Each anchor is a value, a **kind**, and
a provenance string, so no floor or ceiling is a number someone once eyeballed:

- **`ANALYTIC`** — a principled bound: tracking's chance level $\mathbb{E}[\cos\theta]=0$, or
  a true maximum such as "intercept every ball" ($\text{hit\_rate}=1$).
- **`NULL_MEASURED`** — a measured chance floor: the score of a null policy, *run*, not guessed.
- **`REFERENCE_MEASURED`** — a measured ceiling from a named reference agent.

## The random policy is one task-specific floor

The floor is the score of **`:null_random`** — a policy that ignores its input and emits
uniform-random effectors. The policy is reservoir-independent, but its score is specific to
the task, body, actuator mapping, episode length, and randomization. It establishes “no
better than this random-action policy,” not a universal null. `calibrate_task` runs it over
seeds and records the floor with provenance, e.g.
`null=null_random, score_key=forage_score, seeds 0:7, git …`.

This is why several floors sit well above zero: a random forager still drifts $\approx 0.46$
of the way to the source just by wandering, so a forage score only *counts* once agents beat
undirected search.

## Gates keep the indicator honest

An indicator is only trustworthy if a degenerate solution can't game it. The wall task
rewards **collision avoidance** — but "avoid every collision" is trivially won by *not
moving*. So the wall score is collision-free navigation **gated by movement**: a frozen agent
scores $\approx 0$ because the gate — not the score — zeroes it. The gate encodes the
unstated half of the task ("*while behaving*"); it is never the competence signal. (The same
pattern gates rhythm tasks on oscillation amplitude.)

## Descriptors are read alongside a score, never as one

Each task names one **`score_key`** (the capacity indicator) plus any **`descriptor_keys`** —
channels reported next to it but never treated as competence. Collective order parameters
such as **polarization** and **milling** are descriptors in exactly this sense: they measure
*how* a swarm is organised, not whether it succeeded, and reporting one as a "score" would
confuse a quality with a capacity.

## Where the anchors stand today

Current tasks mix analytic and measured floors and use analytic ceilings because no
validated reference agent yet serves as a measured ceiling:

| task | floor | ceiling |
|---|---|---|
| wall | `null` ≈ 0.76 (`nav_score`) | analytic `1.0` — collision-free navigation |
| tracking | analytic `0.0` — chance | analytic `1.0` |
| pong / pong_hitrate | `null` ≈ 0.36 (`hit_rate`) | analytic `1.0` — intercept every ball |
| cartpole (+ swingup) | analytic `0.0` / `null` ≈ 0.16 | analytic `1.0` |
| forage | `null` ≈ 0.46 | analytic `1.0` — agents on source |

A `REFERENCE_MEASURED` ceiling—a genuinely good, *trained* agent as the “1”—is a future
calibration step. Equal ceiling kinds or equal numeric maxima do not make scores from
different tasks commensurate: observation/action semantics, difficulty, gates, and floor
uncertainty differ. Compare models within a declared task; aggregate across tasks only with
an explicit, justified weighting and preparation policy.
