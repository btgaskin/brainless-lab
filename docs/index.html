BrainlessLab.jl

An extensible Julia lab for “brainless” cognition — behaviour that emerges from collectives of simple neuron-like nodes, with no homunculus and no hand-wired control.

The unifying idea is neurons as nodes within a collective: the same node contract holds at every scale. A single-agent task is a Collective of one agent; a dyad is two; a swarm is N. Reservoirs, bodies, tasks, swarm media, recorders, metrics, and Makie visualizations are all wired through lightweight registries, so participants can swap node families or add their own parts without forking the framework.

What it is

BrainlessLab.jl is a summer-institute testbed: a clean framework for running experiments around a settled Falandays baseline. Everything — the neuron model filling a reservoir, the body that turns world state into sensory current and motor output back into action, the task/environment, the swarm medium, the optimizer — is registry-backed. You pick parts by symbol (node=:falandays_base, task=:wall) and the framework assembles them to a matching sensorimotor contract.

Validation means bit-fidelity to the local numpy reference implementations (../v0, ../v0.2) where those reference paths exist. The 2021 paper-faithful baseline is :falandays_base with the original constants; the v0.2 reference itself contains documented experimental departures, so numpy fidelity should not be read as paper fidelity for every platform component.

How it all fits together

Everything is assembled from small, swappable parts. Reading top to bottom, each row takes what you already have, adds one more part, and produces the next unit — the highlighted result of one row is the input to the next:

NodeModel → Reservoir a neuron model fills a reservoir — internally Model ⊗ Connectome ⊗ ConnState ⊗ State
Reservoir + Body → Agent a reservoir in a body — the body owns the morphology (percept → R in, E → motor out)
N × Agent + Medium → Collective{Medium} a population in a world — the body + medium decide the setup: agent–environment (1 agent) or multi-agent (N); one step! drives them all
Collective + Task + Driver → Run a task scores it; a driver (fixed / evolve / plastic) rolls it out into a recorded run

Every box is a registry-backed part: pick it by symbol (node=:falandays_base, task=:wall) or register your own, and the framework wires a matching sensorimotor contract. The sections below open up each part in turn.

Stable baseline vs. experimental

Stable baseline

:falandays_base (alias :falandays) is the settled, paper-faithful 2021 Falandays homeostatic spiking reservoir with its exact constants, run on the 2024 case-study task suite implemented here. It is validated byte-for-byte against the numpy reference. Use it when you need the known reference model, not a new experiment.

Experimental platform

Everything else — the Falandays variants beyond base, the compartmental/CTRNN nodes, the evolution layer, and the swarm/VEN extensions — is the platform built around that baseline. These are useful testbed surfaces for tinkering, but they should not be described as the 2021 paper model.

The core abstraction

A reservoir is exactly these four things:

immutable — set once, at build time mutable — changes every tick
ReservoirInstance{ Model, Connectome, ConnState, State }
Model
the neuron dynamics & params — leak, thresholds, drive, learning rule
Connectome
the wiring structure — who connects to whom (mask, node positions, delays)
ConnState
live connection state — the learned weight matrix, the spike-history buffer
State
live neuron state — membrane activations, targets, spikes, errors

Each step! reads the two immutable parts (teal) to advance the two mutable parts (amber). Swap any one part — a different connectome, a different model — and everything else is reused unchanged.

In code this is literally the type parameter list:

ReservoirInstance{Model, Connectome, ConnState, State}

Both node families in the package share this exact shape. The Falandays family instantiates it as FalandaysModel ⊗ a connectome (DenseConnectome, SpatialConnectome{D}, or DelayedConnectome{D}) ⊗ FalandaysConnState (learned weight matrix + optional spike history) ⊗ FalandaysNeuronState (membrane activations, targets, spikes, errors, noise). The compartmental/CTRNN family instantiates the identical generic with a CompartmentalModel, a Wiring connectome, an (empty) CompartmentalConnState, and a CompartmentalNeuronState (dendrite/soma/hillock voltages). Same shape, different filling.

Connectome types plug into the shared reservoir dynamics purely through dispatch, and three cross-cutting Holy trait axes describe what a connectome supports:

Cross-cutting traits
traitvaluesmeaning
plasticityNoPlasticity / OnlinePlasticitydoes the connectome learn during rollout (Falandays) or is it fixed/evolved (compartmental)
spatialityAspatial / Embedded{D}whether nodes carry positions in a D-dimensional metric space
delaykindUnitDelay / HeteroDelaywhether every edge fires with a single-tick delay or per-edge conduction delays

The sensorimotor contract

A body-owned Morphology (Port / PortSpec) owns encoding and decoding, and is the authoritative source of (n_receptors, n_effectors). The reservoir is built to match — not the other way around. Each tick, for every agent:

Body — encodes / decodes Reservoir — the controller
perceptworld state
→
receptors→ R
→
step!→ spikes
→
effectors→ E
→
motor→ actuation
↶ actuation changes the world, which becomes the next tick’s percept — the loop closes, every tick, for every agent.

PassthroughBody relays a task environment’s R/E directly — used by every single-agent task. VENBody is the embodied swarm agent: it manufactures a 64-channel receptor vector from bearing vision and turns a 3-channel effector vector into VEN kinematics. The controller (the reservoir) is task-agnostic; it is simply built to whatever (R, E) the body/task pair reports.

Effector semantics are intentionally not uniform across tasks — effector channel 1 in Pong is not the same physical quantity as effector channel 1 in cartpole or torus. See docs/contracts.md before comparing raw scores across tasks.

Node families & variants

Every node satisfies the same contract — step!(r, R) -> spikes, effectors(r, spikes) -> E, reset!, n_receptors, n_effectors — so any registered node can be built for any registered task/body dimensions. variants() lists what is currently registered.

Two families, one contract

The Falandays family is a leaky integrate-and-fire reservoir with online homeostatic learning during the rollout: per-node target T (floor 1), firing threshold T′ = 2T, leak 0.25. Because it self-organizes online, it is fair to run untrained — default params and random wiring per seed is a meaningful test.

The compartmental/CTRNN family is a BrainlessLab construction (not a Falandays paper model): dendrite → soma → hillock cells with emergent/evolved weights and no online learning. Because the weights don’t adapt, an untrained compartmental node is random/meaningless — it must be evolved before it is a fair test. See docs/evolution.md.

Tasks & I/O mappings

Every task defines a sensorimotor contract: how many receptors (R) and effectors (E) it needs, how the world is encoded into R, and how E is decoded back into action. A node is built to match (n_receptors, n_effectors); the controller itself is task-agnostic.

Spatial, hemispheric & delayed reservoirs

Three of the newer Falandays variants push node identity beyond “a row/column in a weight matrix” into an embedded, structured, or temporally-realistic wiring space:

:falandays_spatial

Nodes get random positions in a MetricSpace{D}. Recurrent edges are drawn with probability set by an ExpKernel distance falloff (p0 · exp(−dist/λ)) instead of uniform Bernoulli wiring — connectivity reflects geometry.

:falandays_hemispheric

Builds two mirror-placed regions (left/right) via build_hemispheric_connectome. Sensor/effector wiring is contralateral by default; callosum_density sets the rate of bidirectional homotopic cross-region links between mirrored node pairs (0.0 = isolated hemispheres, higher = more coupled).

:falandays_delayed

Per-edge integer conduction delays computed from node distance and a conduction_velocity, carried through a SpikeHistory ring buffer (DelayedConnectome). velocity=Inf collapses every delay to 1, making it numerically identical to the dense reservoir.

DenseConnectome SpatialConnectome{D} DelayedConnectome{D}
→
same reservoir, same dynamics
ReservoirInstance{ FalandaysModel, … }
Connectome slot — drop in any of the three

All three share the same FalandaysModel dynamics and simply supply a different Connectome — the spatiality/delaykind traits let the shared step!/learn_connectome! machinery dispatch to the right recurrent-input and learning rule without duplicating the neuron model.

Both run through one identical tick — observe → step! → actuate → commit. A single-agent task is simply a Collective of one; you assemble either setup by choosing the body + medium pair.

# single agent
simulate(:wall; node=:falandays_base, ticks=300)

# dyad
simulate(:torus; node=:falandays_base, n_agents=2)

# swarm, interactive (needs GLMakie)
explore(:torus; node=:falandays_base, n_agents=6)

In the multi-agent setup, TorusMedium is a periodic 2-D world and VENBody the embodied agent. By default there is no explicit interaction term — agents influence each other purely by being seen, so sensor geometry is the interaction topology. Collective behaviour is read through swarm metrics rather than a single normalized score: polarization (heading alignment), milling (rotational order about the centroid), mean nearest-neighbour/pairwise distance, input stability, and a liveness sanity check on population firing rate.

Extend it

Every surface has a registry: register a part under a symbol, then reference that symbol from high-level code — register_node!, register_task!, register_drive!, register_body!, register_metric!, register_view!, register_ablation!, register_optimizer!.

The headline for extenders: define a NodeModel plus a step! method (optionally a Connectome) and register it — nothing else.

Extending a node means adding methods to the package generics, so import them — otherwise simulate will not see your step!:

import BrainlessLab: step!, effectors, n_receptors, n_effectors

struct MyNode <: Reservoir
    n_receptors::Int
    n_effectors::Int
    spikes::Vector{Float64}
end

MyNode(n_nodes, n_receptors, n_effectors; seed=0) =
    MyNode(n_receptors, n_effectors, zeros(Float64, n_nodes))

function step!(r::MyNode, receptors)
    r.spikes .= maximum(Float64.(receptors)) > 0.5
    return copy(r.spikes)
end

effectors(r::MyNode, spikes) = fill(sum(spikes) / length(spikes), r.n_effectors)
n_receptors(r::MyNode) = r.n_receptors
n_effectors(r::MyNode) = r.n_effectors

register_node!(:mynode, MyNode)
simulate(:wall; node=:mynode, ticks=100)

The same register_*! pattern applies to tasks (register_task! + a TaskSpec), drives (subtype Drive + apply_drive!), bodies, metrics, ablations/interventions, views, and optimizers (subtype AbstractEvolutionStrategy with ask/tell!/result). See docs/contracts.md for the full node/extension contract, including the pack_params/snapshot_state split between evolvable genome and transient runtime state.

Visualization

Visualization is a clean optional layer. The compute core never imports Makie: an engine-side Recorder (channel-gated, downsampling, off the hot path) is the only coupling, and a Makie package extension (BrainlessLabMakieExt) supplies the plotting methods once a Makie backend is loaded.

callwhat it gives you
simulate(task; ...)headless run → a SimResult
visualize(sim)static CairoMakie figure: spike raster + rate + trajectory panels
explore(task; ...)interactive GLMakie window — Play / Step / speed controls
animate(sim; path=...)a behaviour GIF/MP4 of the actual task (eye chasing the stimulus, ball + paddle, cart + pole, agent(s) moving) with a synced firing-rate marker
replay(sim) / replay(rundir)re-render an in-memory result, or reload a saved run directory back into one

Individual recipes (rasterplot, rateplot, trajectoryplot, swarmplot, networkplot, driftplot, fitnessplot) are registered views you can compose directly.

Quick start

using BrainlessLab, CairoMakie

sim = simulate(:wall; node=:falandays_base, ticks=300)  # stable baseline, 1 agent
visualize(sim)

explore(:torus; node=:falandays_base, n_agents=6)        # interactive swarm (needs GLMakie)

The compute core does not depend on Makie — simulate runs headless; plotting methods load once you using CairoMakie (static) or GLMakie (interactive).

Turnkey demo runner

demo/run.jl gives you behaviour GIFs and archived run directories without writing any Julia:

cd demo
julia --project=. run.jl --list                       # list tasks and node variants
julia --project=. run.jl wall --save                   # archive figure + GIF + config
julia --project=. run.jl pong --node falandays_oosawa --save
julia --project=. run.jl torus --n-agents 6 --save     # multi-agent swarm

Statistical benchmark

bench/ is a self-contained sibling project for comparing registered neuron variants across registered tasks: auto-prep by plasticity (Falandays untrained, compartmental trained from a stored genome), best/representative/worst behaviour GIFs, nonparametric statistics with power analysis, and a structured, provenance-tracked output directory.

cd bench
julia --project=. run.jl --neurons falandays_base,compartmental_structured \
                          --tasks wall,pong --no-gifs
